# 00 · 总体方案与技术路线

## 一、项目定位

writeAgent 是一个**面向学术论文写作的智能 Agent**，目标是把研究者写论文时涉及的"需求结构化 → 文献梳理 → 大纲设计 → 正文撰写 → 格式校验 → 语言润色"六个环节，自动化为一条可控、可复现、可部署的流水线。

项目同时满足两类使用场景：

1. **本地独立模式**（开发 / 调试 / 学术演示）：通过 `python -m agent run` 驱动本地 ReAct + Sub-agent 调试链路。
2. **OpenClaw 平台模式**（最终部署）：把 `./skills/` 直接放到龙虾平台的工作区 Skill 目录，由 OpenClaw 自带的 ReAct 大脑按 `SKILL.md.description` 自主调度。

两条路径共享同一份 Skill 实现与同一份 JSON Schema 契约，确保"本地能跑通"等价于"平台能跑通"。

## 二、核心设计原则


| 原则            | 落地方式                                                                                                      |
| ------------- | --------------------------------------------------------------------------------------------------------- |
| 单一事实源         | 所有跨 Skill 中间结果集中存于 `.writeagent/state.json`                                                               |
| 契约先行          | 6 份 [JSON Schema](../schemas/) 定义跨 Skill 字段，pydantic 镜像做运行时校验                                             |
| Skill 自包含     | 每个 Skill 是一个目录（`SKILL.md` + `prompts/` + `scripts/` + `references/` + `assets/`），无 LangGraph/LangChain 依赖 |
| 编排层与执行层解耦     | LangGraph 只在 `agent/` 出现，Skills 严禁 import LangGraph/LangChain                                             |
| 渐进式披露         | 每个 `SKILL.md` 正文 < 500 行，细节写在 `references/` 里按需读取                                                         |
| 双输出           | 每个 Skill 同时产出机器消费的 JSON（写入 state）与人类阅读的 Markdown（写入 `outputs/`）                                           |
| LLM Gateway治理 | 所有模型调用集中在 `agent/llm_gateway.py`，由 Main Agent 或 Sub-agent 发起，Skill script 不直接调用 LLM                       |


## 三、技术路线

### 1. 语言与依赖分层

- **Python 3.10+**：学术工具链最齐全（`bibtexparser` / `pypdf` / `python-docx`）
- **依赖分层**：
  - `[requirements-core.txt](../requirements-core.txt)` —— Skill 端确定性脚本依赖（`pydantic`、`bibtexparser`、`pypdf`、`pyyaml`、`jinja2`、`rich`、`jsonschema`、`python-dotenv` 等）
  - `[requirements-orchestrator.txt](../requirements-orchestrator.txt)` —— 独立 Agent 模式增量依赖（`langgraph`、`langchain-core`、`typer`、模型客户端等）
- 部署到 OpenClaw 时只装 core，零 LangGraph 占用。

### 2. LLM 接入

- 全部走 `agent/llm_gateway.py`，支持 **OpenAI 兼容协议**（自定义 `base_url`）。
- 默认目标：通义千问 DashScope `qwen-plus`；可无改动切换到智谱 GLM-4、DeepSeek、OpenAI 等。
- 三个环境变量：`WRITEAGENT_LLM_API_KEY` / `_BASE_URL` / `_MODEL`。
- 离线模式：`WRITEAGENT_MOCK_LLM=1` 或不设置 `_API_KEY` 时由 Gateway 启用 mock payload。

### 3. 中间结果格式

- **JSON**（机器消费 + Schema 校验）：`state.json` + 各 Skill 内部传递的字段。
- **Markdown**（人类阅读 + LLM 解读）：`outputs/01-...md`、`outputs/02-...md` 等。
- 双格式落地的好处：JSON 用于下游 Skill 编程消费、Schema 校验；Markdown 用于审阅、汇报、文档拼接。

### 4. Schema 契约

6 份核心 JSON Schema（详见 [docs/03](./03-统一输入输出字段规范.md)）：

```
schemas/
├── state.schema.json                 # 顶层 WriteAgentState
├── writing_task.schema.json          # Skill 1 输出
├── literature_report.schema.json     # Skill 2 输出
├── paper_outline.schema.json         # Skill 3 输出
├── paper_draft.schema.json           # Skill 4 输出
├── format_report.schema.json         # Skill 5 输出
└── polish_report.schema.json         # Skill 6 输出
```

`skills/_shared/schemas.py` 提供 pydantic 镜像，每个 Skill 在产出前都做一次 `model_validate`，确保下游能无障碍消费。


## 四、后续接口约定

1. Skill 3 / Skill 4，按 `skills/{paper-outline, paper-content-generation}/SKILL.md` 的"输入/输出/必须填充字段"约定实现 `scripts/run.py`。
2. Skill 5 / Skill 6，按对应 `SKILL.md` 实现并产出 docx/PDF 终稿。
3. Skill 3-6 必须沿用 `skills/_shared/` 中的确定性工具（io、schemas），禁止在 Skill 内新增模型调用入口；如需新增共享能力，统一在 `_shared/` 下扩展。
4. Skill 3-6 必须遵循 [docs/05-Skill开发规范.md](./05-Skill开发规范.md)，由 Sub-agent 写 `state.intermediate.*`，由 Skill script 写正式产物。
5. 任何对 Schema 的修改先在 `schemas/*.schema.json` 提案并 PR，再同步更新`skills/_shared/schemas.py`，避免 6 个模块同时打架。
